# 02 — Data Snapshot vs Container Checkpoint 성능 비교

## 실험 목적

본 실험은 Redis의 기본 영속화 방식인 `SAVE`, `BGSAVE`와 제안한 
**MDB-KCP(Container Checkpoint 기반 Persistence Framework)** 를 비교하여 각 방식의 **체크포인트 생성 시간, 서비스 다운타임, 복구 시간**을 분석하는 것을 목적으로 한다.

단순한 저장 성능 비교가 아니라, 운영 환경에서의 **가용성 및 복구 전략(RTO 관점)** 을 중심으로 평가하였다.


## 비교 대상 Persistence 전략

### 1. SAVE

- Redis 메인 프로세스가 직접 RDB 스냅샷을 생성
- 스냅샷 생성이 완료될 때까지 모든 요청을 차단

**특징**
- 구현이 단순함
- 스냅샷 생성 중 서비스가 완전히 중단됨
- 다운타임이 가장 큼

### 2. BGSAVE

- `fork()`를 통해 자식 프로세스를 생성
- 자식 프로세스가 백그라운드에서 RDB 파일 생성
- 메인 프로세스는 요청 처리 지속

**특징**
- 서비스는 유지됨
- Copy-on-Write로 인한 메모리 오버헤드 발생

### 3. MDB-KCP

- CRIU를 활용하여 컨테이너 전체 상태를 체크포인트
- 메모리, 파일 디스크립터, 네임스페이스 상태까지 저장
- 체크포인트 결과를 TAR로 생성 후 OCI 이미지로 변환

**특징**
- 프로세스 상태를 포함한 컨테이너 단위 스냅샷
- 이미지 변환 과정으로 인해 생성 시간은 상대적으로 길 수 있음
- 복구 시 데이터 로딩 과정이 필요 없음

---

## 실험 결과 분석

### 1. 체크포인트 생성 시간

- `SAVE`와 `BGSAVE`는 비교적 짧은 생성 시간을 보임
- `MDB-KCP`는 TAR 생성 및 OCI 이미지 변환 과정으로 인해 상대적으로 더 긴 시간이 소요됨

그러나 MDB-KCP는 단순 데이터 파일이 아닌  **프로세스 전체 상태를 저장한다는 점에서 본질적인 차이가 존재한다.**

---

### 2. 서비스 다운타임

- `SAVE`  
  - 스냅샷 생성 동안 서비스가 완전히 정지  
  - 가장 긴 다운타임 발생

- `BGSAVE`  
  - 서비스는 유지되지만 fork 시 리소스 오버헤드 존재

- `MDB-KCP`  
  - 메모리 dump 구간에서만 짧은 일시 정지 발생  
  - 전체 생성 시간이 아니라, 실제 서비스 중단 시간은 상대적으로 짧음

---

### 3. 복구 시간 (Restore Time)

- `SAVE` / `BGSAVE`  
  - 컨테이너 재시작
  - RDB 파일 로드
  - 데이터 재구성 과정 필요
  → 복구 시간이 상대적으로 김

- `MDB-KCP`  
  - OCI 이미지 기반 컨테이너 실행
  - 메모리 상태를 그대로 복원
  → 데이터 로딩 과정이 필요 없음

실험 결과, MDB-KCP 방식은 기존 RDB 기반 방식 대비  
**최대 11.3배 빠른 복구 속도**를 보였다.

---

## 결론

본 실험은 단순 저장 속도 비교가 아닌, 각 영속성 유지 전략의 운영 관점 특성을 분석한 결과이다.

- `SAVE`는 안정적이나 다운타임이 큼
- `BGSAVE`는 서비스 중단은 적으나 메모리 오버헤드 존재
- `MDB-KCP`는 생성 비용은 증가하지만 복구 속도가 매우 빠름

즉, MDB-KCP는 **복구 시간(RTO) 단축이 중요한 환경에서 효과적인 전략**임을 확인하였다.
